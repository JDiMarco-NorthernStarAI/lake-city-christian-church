
Create a "Sign Ups" system that connects events/opportunities to forms, allows users to see what they and others have signed up for, and gives admins control over post-submission visibility. This builds on the existing form builder and events system.

## OVERVIEW

Create a new "Sign Ups" section that serves as a hub for all church sign-up opportunities. Admins create "Sign Up Events" (different from calendar events) that display with an image, description, and details. Each Sign Up Event links to a form. After submission, users see configurable confirmation content based on admin settings.

## DATABASE SCHEMA

### Sign Up Events Table
- id (primary key, UUID)
- title (string, required - "VBS 2025 Registration", "Easter Potluck Sign Up", "Serve Team Volunteers")
- slug (string, unique, URL-friendly - auto-generated from title, editable)
- description (text - rich text/markdown supported)
- image_url (string, nullable - hero image for the event)
- thumbnail_url (string, nullable - smaller version for cards)
- form_id (foreign key to Forms, required)
- category (enum: 'kids_ministry', 'student_ministry', 'small_group', 'fellowship', 'volunteer', 'class', 'event', 'other')
- status (enum: 'draft', 'published', 'closed', 'archived')
- visibility (enum: 'public', 'members_only', 'unlisted')
- signup_start_date (timestamp, nullable - when signups open)
- signup_end_date (timestamp, nullable - when signups close)
- event_date (timestamp, nullable - when the actual event occurs, for display)
- event_end_date (timestamp, nullable - for multi-day events)
- location (string, nullable)
- max_signups (integer, nullable - null for unlimited)
- current_signup_count (integer, default 0 - cached count)
- waitlist_enabled (boolean, default false)
- waitlist_count (integer, default 0)
- cost (decimal, nullable - for display, actual payment handled separately if needed)
- contact_name (string, nullable - "Questions? Contact...")
- contact_email (string, nullable)
- contact_phone (string, nullable)
- post_submission_settings (JSON - see schema below)
- created_by (foreign key to Users)
- created_at (timestamp)
- updated_at (timestamp)
- deleted_at (timestamp, nullable - soft delete)

### Post-Submission Settings JSON Schema (stored in post_submission_settings column):
{
  "display_type": "thank_you" | "summary_own" | "summary_all" | "summary_all_anonymous" | "redirect" | "custom",
  "thank_you_title": "You're Signed Up!",
  "thank_you_message": "Thanks for signing up! We're excited to see you there.",
  "redirect_url": null,
  "redirect_delay_seconds": 3,
  "show_signup_number": true,
  "show_event_details": true,
  "show_add_to_calendar": true,
  "show_share_button": false,
  "summary_fields": null,
  "summary_grouping_field": null,
  "allow_edit_submission": false,
  "allow_cancel_signup": true,
  "confirmation_email": true,
  "reminder_enabled": false,
  "reminder_hours_before": 24
}

### Post Submission Display Types Explained:
- 'thank_you': Show custom thank you message only
- 'summary_own': Show thank you + user's own submission details
- 'summary_all': Show thank you + ALL submissions with names (great for potlucks, volunteer slots)
- 'summary_all_anonymous': Show thank you + all submissions WITHOUT names (just see what's taken)
- 'redirect': Redirect to external URL after brief thank you
- 'custom': Show custom HTML/content block

### Sign Up Event Settings Table
- signup_event_id (primary key, foreign key to Sign Up Events)
- require_login (boolean, default false - must be logged in to sign up)
- one_signup_per_user (boolean, default true - prevent duplicate signups)
- one_signup_per_household (boolean, default false - based on shared address/email)
- show_spots_remaining (boolean, default true - "5 spots left!")
- show_whos_coming (boolean, default false - list of names on the event page before signup)
- allow_guest_signups (boolean, default false - can add family members/guests in same form)
- notify_on_signup (JSON array of emails - who gets notified on each signup)
- notify_on_full (JSON array of emails - who gets notified when full)
- auto_close_when_full (boolean, default true)
- tags (JSON array - for filtering: ["family-friendly", "food", "outdoor"])
- created_at (timestamp)
- updated_at (timestamp)

### Sign Up Submissions Table
- id (primary key, UUID)
- signup_event_id (foreign key to Sign Up Events)
- form_submission_id (foreign key to Form Submissions)
- user_id (foreign key to Users, nullable for guest signups)
- signup_number (integer - their place in line, e.g., "You are #15")
- status (enum: 'confirmed', 'waitlisted', 'cancelled', 'no_show', 'attended')
- waitlist_position (integer, nullable)
- promoted_from_waitlist_at (timestamp, nullable)
- cancelled_at (timestamp, nullable)
- cancellation_reason (text, nullable)
- checked_in (boolean, default false)
- checked_in_at (timestamp, nullable)
- checked_in_by (foreign key to Users, nullable)
- guest_count (integer, default 0 - additional guests beyond primary signup)
- admin_notes (text, nullable)
- created_at (timestamp)
- updated_at (timestamp)

## API ENDPOINTS

All endpoints under /api/v1/

### Public Sign Up Events
- GET /signups - List all published sign up events (with filters: category, upcoming, has_spots)
- GET /signups/featured - Get featured/highlighted sign up events for homepage
- GET /signups/:slug - Get sign up event details by slug
- GET /signups/:slug/availability - Get current signup count, spots remaining, waitlist status
- GET /signups/:slug/preview - Get summary of current signups (respects visibility settings)

### Sign Up Submission
- POST /signups/:slug/submit - Submit signup (processes form, creates signup record)
- GET /signups/:slug/my-submission - Get current user's submission for this event
- PUT /signups/:slug/my-submission - Update submission (if allowed)
- DELETE /signups/:slug/my-submission - Cancel signup (if allowed)
- GET /signups/:slug/confirmation - Get confirmation/summary page data

### User's Sign Ups
- GET /my/signups - Get all of current user's signups (past and upcoming)
- GET /my/signups/upcoming - Get only upcoming signups
- GET /my/signups/:id - Get specific signup details

### Admin Sign Up Event Management
- GET /admin/signups - List all sign up events (all statuses)
- GET /admin/signups/:id - Get sign up event with full details and stats
- POST /admin/signups - Create new sign up event
- PUT /admin/signups/:id - Update sign up event
- DELETE /admin/signups/:id - Delete sign up event (soft delete, preserves submissions)
- POST /admin/signups/:id/duplicate - Duplicate event (for recurring events)
- PUT /admin/signups/:id/status - Change status (publish, close, archive)
- POST /admin/signups/:id/image - Upload event image

### Admin Submission Management
- GET /admin/signups/:id/submissions - Get all submissions for event (paginated, filterable)
- GET /admin/signups/:id/submissions/export - Export as CSV
- GET /admin/signups/:id/submissions/:subId - Get specific submission
- PUT /admin/signups/:id/submissions/:subId - Update submission status, notes
- DELETE /admin/signups/:id/submissions/:subId - Delete/cancel submission
- POST /admin/signups/:id/submissions/:subId/promote - Promote from waitlist
- POST /admin/signups/:id/check-in/:subId - Check in attendee
- POST /admin/signups/:id/check-in/bulk - Bulk check in by list of IDs

### Admin Waitlist Management
- GET /admin/signups/:id/waitlist - Get waitlist in order
- POST /admin/signups/:id/waitlist/promote-next - Promote next person on waitlist
- POST /admin/signups/:id/waitlist/notify - Send "spot available" notification to waitlist

## INTEGRATION WITH EXISTING SYSTEMS

1. Sign Up Events are separate from calendar Events but can link via event_date for calendar display
2. Forms used for sign ups work normally in the existing form builder, they just get extra "After Submission" options when linked to a Sign Up Event
3. Sign Up Submissions create records in BOTH sign_up_submissions AND the existing form_submissions tables (linked via form_submission_id)
4. The existing form submission viewer should show a link to the Sign Up Event if the submission came from a sign up
5. User's "My Sign Ups" section should be accessible from their profile/dashboard alongside existing "My Forms" or similar
6. Cancellation updates sign_up_submissions.status to 'cancelled' but preserves the form_submission record
7. Waitlist promotions should trigger notification via existing notification system (push and/or SMS if enabled)
8. Confirmation emails use the existing email template system

## MOBILE RESPONSIVE REQUIREMENTS

1. Sign Ups hub page cards stack vertically on mobile
2. Event detail page hero image scales appropriately
3. Form embedding works fully on mobile
4. Confirmation page is touch-friendly with large tap targets
5. "Add to Calendar" generates proper mobile-compatible links
6. My Sign Ups page is easily accessible from mobile navigation
7. Admin pages function on tablet at minimum for on-site check-in
