Add SMS/Text Messaging functionality to the existing notification system using Twilio. This should integrate with the current push notification feature as an additional channel option. Here are the requirements:

## OVERVIEW

Extend the existing notification system to support SMS as a delivery channel alongside push notifications. Admins should be able to choose "Push Notification", "SMS Text", or "Both" when sending messages. The SMS system will use Twilio for delivery.

## ENVIRONMENT VARIABLES NEEDED

Add these to the project secrets:
- TWILIO_ACCOUNT_SID
- TWILIO_AUTH_TOKEN
- TWILIO_PHONE_NUMBER
- TWILIO_WEBHOOK_SECRET

### Twilio Webhooks (public, verified by Twilio signature)
- POST /webhooks/twilio/incoming - Handle incoming SMS messages
- POST /webhooks/twilio/status - Handle delivery status callbacks

### Incoming Message Webhook Handler:
1. Verify Twilio signature using TWILIO_WEBHOOK_SECRET
2. Log the incoming message to sms_incoming_messages table
3. Check for opt-out keywords: STOP, STOPALL, UNSUBSCRIBE, CANCEL, END, QUIT
4. Check for opt-in keywords: START, YES, UNSTOP, SUBSCRIBE
5. If opt-out: add to opt_out table, update user sms_opt_in=false, send opt_out_confirmation
6. If opt-in: remove from opt_out table, update user sms_opt_in=true, send opt_in_confirmation
7. For other messages: mark requires_response=true, optionally notify admins
8. Return TwiML response (empty or auto-reply based on settings)

### Status Webhook Handler:
1. Verify Twilio signature
2. Find recipient record by twilio_message_sid
3. Update status: queued, sent, delivered, undelivered, failed
4. If delivered: set delivered_at timestamp
5. If failed: store error_code and error_message
6. Update message aggregate counts (delivered_count, failed_count)
7. If available, store cost from Twilio

## TWILIO INTEGRATION SERVICE

Create a TwilioService class/module:

### Phone Lookup (on user registration or phone update):
- Use Twilio Lookup API to validate phone number
- Detect phone type (mobile, landline, voip)
- Get carrier information
- Format to E.164 standard
- Only mobile phones can receive SMS

### Send Single SMS:
- Check if phone is opted out first
- Check quiet hours if enabled
- Format message with church prefix if configured
- Add opt-out footer for first message to recipient
- Send via Twilio Messages API
- Store twilio_message_sid for tracking
- Return success/failure

### Send Bulk SMS (for groups):
- Get list of eligible recipients (opted in, mobile phone, not in quiet hours)
- Calculate estimated cost before sending
- Process in batches of 100 to respect rate limits
- Use 100ms delay between messages to avoid throttling
- Track progress and update message status
- Handle personalization (replace variables per recipient)
- Queue processing for large groups (>500 recipients)

### Message Personalization:
Replace these variables in message body:
- {{name}} → user's full name
- {{first_name}} → first word of name
- {{last_name}} → last word of name
- {{church_name}} → "Lake City Christian Church"

### Cost Calculation:
- Base rate: $0.0079 per SMS segment
- Segment = 160 characters (GSM) or 70 characters (Unicode)
- Calculate: Math.ceil(message.length / 160) * recipient_count * 0.0079
- Display estimated cost before send confirmation

### Quiet Hours Check:
- Get current time in configured timezone
- If quiet_hours_enabled and current time is within quiet hours:
  - For immediate send: queue for next available time (quiet_hours_end)
  - For scheduled send: adjust if scheduled during quiet hours
- Return whether message can be sent now or when it will be sent

## COMPLIANCE & BEST PRACTICES

1. Always include church identification in messages
2. First message to any recipient must include opt-out instructions
3. Honor opt-outs immediately (within same webhook request)
4. Never send during quiet hours unless emergency
5. Maintain opt-out records indefinitely
6. Provide easy opt-out management for users in their profile
7. Log all messages for audit trail
8. Rate limit sending to avoid carrier blocking

## ERROR HANDLING

1. Invalid phone numbers: Mark user's phone_type as 'unknown', skip for SMS
2. Twilio API errors: Log, mark message as failed, alert admin
3. Rate limiting: Implement exponential backoff, queue excess messages
4. Delivery failures: Track error codes, provide actionable info in admin UI
5. Webhook failures: Return 200 to Twilio, log error, alert admin
